# YAML-BERT v8 — Design Spec

## Goals & Non-Goals

### Primary goal

Produce **general-purpose document-level vector representations** for individual YAML manifests — analogous to how text embedding models produce general-purpose sentence/document vectors. Vectors should be useful across many downstream tasks (retrieval, clustering, anomaly scoring, drift detection, LLM-agent context compression) without being optimized for any single one.

### Secondary goal

**Retire the compound-target token-level workaround.** v4-v7's `kind::parent::child` trigram targets were a workaround for the lack of a document-level representation that could carry kind context. With a document vector available, per-token prediction can use atomic targets (just the child key, vocab ~1000) conditioned on the doc vector + parent subtree vector.

### Non-goals

- **Multi-document modeling.** Cross-document attention, cluster-as-document, application-level grouping — all deferred to a future version.
- **OpenShift-specific corpus.** v8 architecture must be domain-agnostic; OpenShift specialization happens later via training data, not architecture.
- **Beating frontier LLMs on general code/text understanding.** Not the goal.
- **Improving on v7's specific capability/structural test scores.** v8 changes the prediction format; test compatibility is not preserved.

### Architecture family (fixed)

- Encoder-only transformer (BERT-style)
- MLM-style self-supervised pretraining
- Tree positional encoding retained (`depth` + `sibling_index` + `node_type` embeddings summed into input)

## Architecture Sketch

```
┌──────────────────────────────────────────────────────────────┐
│  Input: linearized YAML nodes                                │
│  (token_id, depth, sibling, node_type, parent_path) per node │
└──────────────────────────────────────────────────────────────┘
                            ↓
┌──────────────────────────────────────────────────────────────┐
│  Encoder (unchanged from v7 family)                          │
│  Transformer + tree positional encoding                      │
│  Output: hidden state h_i per token                          │
└──────────────────────────────────────────────────────────────┘
                            ↓
┌──────────────────────────────────────────────────────────────┐
│  Tree Aggregator (NEW)                                       │
│  Bottom-up per key node v:                                   │
│    s_v = combine(h_v, children_of_v)                         │
│  Output: subtree vector s_v per KEY node                     │
│          doc_vec = root aggregation                          │
└──────────────────────────────────────────────────────────────┘
                            ↓
          ┌─────────────────┴─────────────────┐
          ↓                                   ↓
┌─────────────────────────┐         ┌────────────────────────────┐
│ Token Head (NEW)        │         │ Reconstruction Head (NEW)  │
│ At each masked position │         │ At each masked SUBTREE:    │
│ predicts the atomic     │         │ given doc_vec + position   │
│ child key.              │         │ info, predict the masked   │
│ input: [h_i ; doc_vec ; │         │ subtree's structural       │
│         s_parent]       │         │ contents.                  │
│ output vocab: ~1000     │         │                            │
└─────────────────────────┘         └────────────────────────────┘
          ↓                                   ↓
   atomic MLM loss              reconstruction loss
```

### What's new vs v7

- **Tree Aggregator** producing subtree vectors and a doc vector
- **Atomic Token Head** replacing the compound `simple_head` + `kind_head`. Output vocab drops from ~28K compounds to ~1K atomic keys.
- **Reconstruction Head** providing doc-level training pressure.

### What stays vs v7

- Encoder architecture, masking strategy at token level, vocabulary for input keys/values, tree positional encoding.

### Aggregator details

- **Bottom-up by depth.** Nodes at the deepest level computed first, then their parents, etc.
- **Per-key vectors.** Every KEY node gets a subtree vector. VALUE nodes use their hidden state directly (no separate vector).
- **Vectorization.** Computed level-by-level; within each level, all combines are independent and parallel. ~9 sequential stages (one per depth level given Lever-5 depth cap).
- **Combine function.** Open — to be decided in Phase 0. Candidates: `mean` (no params), attention pool (small learnable), Tree-LSTM (more params). Start with `mean` to isolate "does the aggregation paradigm work" from "is the combine function expressive enough."
- **Children identification.** Computed once per document from `parent_path` strings — same pattern as the `tree_distances` precompute used in the tree-bias experiment.

### Token Head details

- Input: `[h_i ; doc_vec ; s_parent(i)]` where `s_parent(i)` is the subtree vector of the IMMEDIATE parent key of position `i` (one level up in the tree). Three vectors concatenated.
- Output: probability over atomic key vocabulary (~1000 classes).
- Kind discrimination comes from `doc_vec` carrying "this is a Pod" / "this is a Service" information, not from output-vocabulary structure.

### Reconstruction Head details

- Output target is the **structural pattern of the masked subtree**: which keys appear, in what tree shape. (Scalar values within the subtree are NOT predicted — too noisy and not the signal we care about.)
- Simplest viable encoding: **bag of keys present in the masked subtree** (multi-label classification over ~1000 atomic keys with BCE loss). Captures "what's in there" without requiring sequence/tree generation.
- More elaborate variants (predict tree shape, predict token sequences) deferred — start simple, escalate if signal is weak.

## Phase 0 — Prototype & Benchmark (HARD GATE)

v7's tree-bias surprise (~15× training slowdown discovered only at training time) is the reference cautionary tale. We don't commit to the full v8 design until a prototype demonstrates the aggregator pattern is GPU-efficient and the architecture is trainable without obvious instability.

### Scope

Build a minimal end-to-end variant:

1. **Aggregator** with mean combine (no learnable params; cleanest baseline).
2. **Atomic Token Head** with `[h_i ; doc_vec ; s_parent]` conditioning.
3. **No reconstruction head yet** — Phase 0 tests aggregator + atomic prediction interaction only.
4. **Initialize encoder weights from random** (clean test of the v8 architecture). Using v7 weights would conflate "did the new architecture train" with "did v7's pre-existing knowledge transfer."

### Phase 0 success criteria

| Metric | Target |
|---|---|
| Training step time vs v7 quick mode | Within 50% overhead |
| GPU utilization during training | > 70% |
| Memory usage | Fits L4 (24GB) with batch_size=32 |
| Training stability | No NaNs over 1 epoch on 5K docs; loss trending down |
| Atomic prediction validity | Loss curve nontrivially below random baseline (`log(vocab_size)`) by end of epoch 1 |
| Kind discrimination probe | After training, a linear probe on doc_vec can distinguish ≥10 common kinds at > 70% accuracy. (Validates the doc vector actually carries kind info, which is the whole reason for retiring compound targets.) |

### Phase 0 outcomes

- **All pass** → proceed to Phase 1, refining the design with Phase 0 learnings.
- **Some fail with diagnosable cause** → fix specific issue, re-prototype, decide.
- **Multiple unrelated failures** → reconsider whether v8 architecture is the right direction; this is exit criteria for the design.

### Phase 0 time budget

~1 week implementation + ~1 day benchmarking. Faster failure is preferred; if obvious gotchas emerge in the first few hours, stop and replan.

## Phase 1 — Full v8 (subject to Phase 0 outcomes)

After Phase 0 confirms the architecture is viable, Phase 1 fleshes out:

### Training objective

- **Token-level MLM** with atomic targets (carried over from Phase 0)
- **Reconstruction objective** — mask random subtrees and predict the bag of keys present
- **Joint training** with weighted combined loss: `L = α·L_mlm + β·L_reconstruction`
  - Initial weights: `α=1.0, β=0.5`
  - Tune empirically; may need warmup phase with `β=0` for the first 1-2 epochs

### Combine function selection

Pending Phase 0. If mean-combine produces acceptable doc-vec quality, keep it. If it's too crude (poor downstream performance), escalate to attention-based combine and re-measure.

### Reconstruction details

Masking strategy: per document, randomly mask 1-3 subtrees (depth ≥ 1, not the root; each subtree contains a small fraction of nodes — e.g., a single container's spec or a livenessProbe block). Combined with token-level MLM masking, total masked positions target ~15-20%. Token-level masks and subtree masks must not overlap (resolved at corpus-construction time).

Decoder architecture: small MLP or single-layer transformer reading `[doc_vec ; position embedding for the masked subtree root]`, output is a ~1000-class BCE multi-label.

### Training schedule

- 30 epochs on full 276K-doc corpus, same scale as v6.1/v7
- Mixed-precision (fp16) for speed
- num_workers ≥ 4 in DataLoader (lesson from v7)

## Open Implementation Details (Gotchas)

Documented in advance to acknowledge they need solutions; not all have to be solved before Phase 0.

1. **Padding for variable-shape trees.** Batches contain docs with different node counts, tree shapes, and children counts per parent. Decision: pad to max-nodes-in-batch + scatter ops, OR use ragged tensors via PyTorch Geometric / DGL. Phase 0 baseline: padded scatter (simpler code).
2. **Variable depth across batch.** Max depth differs per doc. Depth-loop bookkeeping needs per-doc masking.
3. **Circular dependency in subtree reconstruction.** Masked-subtree tokens still get hidden states from encoder (model's prediction-from-context); these flow into doc_vec, which is then used to predict the masked content. Subtle leak. Solutions: (a) exclude masked positions from aggregator inputs; (b) two-pass encoding (expensive); (c) accept the soft leak as long as performance is OK. Empirical question — settled in Phase 1.
4. **List-index handling in children lookup.** parent_paths include numeric indices (`spec.containers.0`); children-of-parent must treat list items correctly.
5. **Backprop through scatter ops.** PyTorch supports native scatter ops; custom scatter_softmax (for attention combine) may interact poorly with `torch.compile`. Verify before relying.
6. **Reconstruction-head output design.** Bag-of-keys is the chosen starting point. If signal is weak, consider predicting tree shape (more complex output) or token sequences (most complex).
7. **Joint-loss weighting.** Different loss scales between MLM (~28K-vocab cross-entropy in v7; ~1K in v8) and reconstruction (BCE multi-label) need balancing. Empirical tuning.
8. **Token-mask vs subtree-mask overlap.** When both mask types are active, overlaps create unclear training signals. Decision: make masks mutually exclusive at corpus-construction time, OR ensure overlap handling is well-defined.
9. **Doc-vector usage at training time.** Token head reads `doc_vec`, which is computed from all token hidden states INCLUDING the masked-position's hidden state. Subtle: the masked position's "context for itself" includes the aggregator's view of the broader doc. Probably fine but should be measured.

## Evaluation Framework

### Primary metrics (Phase 0 gate)

- Aggregator perf vs v7 baseline (see Phase 0 criteria)
- Linear probe accuracy on doc_vec for kind classification (test that atomic + doc_vec conditioning actually learned kind discrimination)
- Loss trajectory (must converge, no NaNs)

### Secondary metrics (Phase 1 evaluation suite)

Following the text-embedding-model evaluation paradigm (MTEB-inspired), construct a multi-task evaluation suite. Each task evaluated independently; average score is the "v8 quality" metric.

**YAML-MTEB candidates (to be finalized in Phase 1):**

| Task | Type | How to construct |
|---|---|---|
| Kind classification | Probe | Linear classifier on doc_vec; predict kind |
| Has-feature classification | Multi-probe | Per feature (has_securityContext, has_resourceLimits, has_livenessProbe, etc.), linear probe |
| Within-kind retrieval | Retrieval | For Pods only, hold out N pods; for each query, retrieve K nearest by doc_vec cosine; eval by structural similarity heuristic |
| Drift sensitivity | Synthetic | Take a doc, perturb (add field / scramble label / change replica count); measure vector shift magnitude; expect small for cosmetic changes, larger for semantic changes |
| Subtree retrieval (if subtree vectors used) | Retrieval | Same as within-kind but at subtree level (e.g., find similar `livenessProbe.httpGet` configurations) |
| Reconstruction quality | Aux | If reconstruction head exists, evaluate held-out subtree reconstruction precision/recall on bag-of-keys |

**Baselines for comparison:**

- `mean-pool(v7 token hidden states)` — null hypothesis: does v8 beat free pooling of v7?
- `random-vector baseline` — sanity check

If v8 doesn't significantly beat `mean-pool(v7)` on the suite, the additional complexity isn't justified.

### Evaluation construction time

- ~3 days to build the YAML-MTEB suite, parallel to Phase 1 implementation
- Use held-out 10% of training corpus as eval set

## Decision Gates

| Gate | Condition | Action if not met |
|---|---|---|
| Phase 0 perf | Aggregator overhead < 50% | Optimize (custom kernels, sparse ops); if still slow, reconsider |
| Phase 0 stability | No NaNs, loss converges | Debug; if instability persists, switch combine to mean, re-test |
| Phase 0 kind probe | > 70% kind accuracy from doc_vec | Try attention combine; if still poor, consider retaining compound targets as fallback |
| Phase 1 vs `mean-pool(v7)` | v8 wins on ≥ 4 of 6 eval tasks | Iterate on objective / architecture; if no clear win, v8 not justified |
